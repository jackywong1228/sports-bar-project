## 背景

本次改造替代废弃的闸机数据源，会员到店统一由前台扫会员动态二维码完成识别+核销+打卡。

- **小程序端**：新增"我的会员码"页面，生成 30 秒短期 JWT 二维码，每 25 秒自动刷新
- **管理后台**：扫码核销页扩展 `MEMBER:` 协议支持，增加场馆下拉、会员卡片、散客到店流程
- **后端**：新增 4 个接口，复用 `GateCheckRecord`（gate_id=NULL 标记前台扫码）

涉及提交：
- `6021dd0` feat: 会员动态二维码 + 前台扫码核销 + 散客邀请打卡
- `97b2588` fix: 修复会员二维码页面 WXML 编译错误

## 阶段 1: 冒烟测试（不依赖小程序）

**目标**：验证后台 Verify 页改造没有破坏旧功能，新 UI 元素渲染正常。

### 步骤 1.1 iPad Safari 登录后台

1. Safari 打开 https://yunlifang.cloud
2. 用管理员账号登录
3. 左侧菜单 → **票券管理** → **扫码核销**
   - 对应路由：`/coupon/verify`
   - （也可以直接访问 `https://yunlifang.cloud/staff/scan`，是同一个组件的快捷入口）

**通过标准**：页面正常打开，无白屏，无红色错误提示

**失败排查**：开 Safari 远程调试（或 Chrome DevTools），看 Console 有没有 JS 报错

### 步骤 1.2 场馆下拉渲染

1. 页面顶部应该有一个**"当前服务场馆"**的下拉框（本次新加）
2. 点开下拉框 → 应能看到所有场馆列表
3. 随便选一个场馆（比如"1号包厢"）

**通过标准**：下拉框能展开，能选中，选中后显示场馆名

### 步骤 1.3 localStorage 持久化

1. 选完场馆后，按 F5 / 下拉刷新整个页面
2. 重新加载后观察下拉框

**通过标准**：刷新后下拉框里**依然保留刚才选的场馆**（不是空的）

### 步骤 1.4 旧优惠券核销流程不回归

1. 找一个会员已持有的未用优惠券 → 在后台复制它的核销码或 QR 内容
2. 在扫码核销页面**手动输入**或扫一个 `COUPON_VERIFY:xxx` 格式的 QR
3. 应弹出旧的确认对话框 → 确认后核销成功

**通过标准**：旧流程完全正常

### 步骤 1.5 旧预约核销流程不回归

1. 在扫码核销页面**手动输入框**输入一个预约编号（如 `RES20260407001`）
2. 应弹出旧的确认对话框 → 确认后核销成功

**通过标准**：旧流程完全正常

> **如果阶段 1 有任何失败，先停下来排查。阶段 2 以上不要开始。**

## 阶段 2: 小程序端（不依赖后台）

**目标**：验证会员能拿到动态二维码。

### 步骤 2.1 进入我的会员码页面

1. 会员 A 用手机微信打开**体验版小程序**
2. 确认已登录（如果没登录就走登录流程）
3. 点底部 Tab → "我的"
4. 在菜单列表里应该看到 **"我的会员码"** 这一项
5. 点进去

**通过标准**：页面正常打开，显示会员卡片 + 一个大方形二维码

**失败排查**：
- 如果报 WXML 错误，说明你没有上传最新版本，检查体验版版本号是不是 1.1.0
- 如果报"请先登录"，检查会员 A 是否已在小程序登录

### 步骤 2.2 二维码渲染

1. 观察二维码是否清晰，有没有乱码或空白
2. 头像位置：如果会员有头像应显示头像，没有应显示姓的第一个字

**通过标准**：二维码渲染清晰，颜色按会员等级匹配：
- S 级：墨绿色 `#1A5D3A`
- SS 级：金色 `#C9A962`
- SSS 级：深金色 `#8B7355`

### 步骤 2.3 倒计时和自动刷新

1. 观察下方的**倒计时进度条**
2. 进度条应该**平滑地从 100% 递减到 0%**
3. 倒计时数字从 "30秒" 递减到 "0秒"
4. 到 25 秒时应该**自动刷新**一次（二维码图像会变），进度条回到 100%

**通过标准**：每 25 秒自动刷新，进度条平滑，无卡顿

### 步骤 2.4 切后台再回来

1. 按 Home 键切到微信主界面（或按返回）
2. 等 5 秒
3. 重新点进"我的会员码"

**通过标准**：二维码正常显示，倒计时重新从 30 秒开始

> **阶段 2 通过后再继续阶段 3。**

## 阶段 3: 端到端核心流程（有预约的会员）

**目标**：验证最关键的"扫码 → 识别 → 核销 → 打卡"链路。

### 步骤 3.1 员工扫会员 A 的码

1. 手机 A 保持打开"我的会员码"页面
2. 用**另一台设备（iPad Safari 或 PC）**登录后台 → 扫码核销页
3. 确认顶部场馆下拉已选中一个场馆
4. 点 **"扫一扫"**按钮
5. 第一次会弹摄像头权限 → **允许**
6. 把摄像头对准手机 A 的二维码

**通过标准**：
- 不弹出旧的"确认核销"对话框
- 页面上出现一个新卡片，显示：
  - 会员 A 的头像 + 昵称 + 等级标签
  - "今日待核销 (1)" 标题
  - 1 条预约卡片（场地名 / 时间段 / 教练 / 价格）
  - 预约卡片右侧有绿色 **[核销]** 按钮

**失败排查**：
- `HTTP 410 二维码已过期`：扫得太慢（token 只有 30s）或两端时间不同步，重试
- `HTTP 410 二维码无效`：扫到的不是 MEMBER 协议，检查小程序 refreshQrCode 逻辑
- 卡片空白：检查 `current_venue_id` 是否传了，看 Network 响应

### 步骤 3.2 核销预约

1. 点预约卡片上的 **[核销]** 按钮
2. 应弹出二次确认对话框
3. 确认 → 应 toast "核销成功 +X 积分"
4. 该预约卡片**从列表中消失**

**通过标准**：核销成功，预约从列表消失

### 步骤 3.3 验证数据库（关键）

在服务器上直接查数据库：

```bash
ssh root@yunlifang.cloud "mysql -u root sports_bar -e \"
SELECT id, reservation_no, status, is_verified, verified_at, verified_by
FROM reservation
WHERE member_id=<会员A的ID> AND reservation_date=CURDATE()
ORDER BY id DESC LIMIT 3;

SELECT id, member_id, venue_id, gate_id, check_date, check_in_time, duration, points_earned
FROM gate_check_record
WHERE member_id=<会员A的ID>
ORDER BY id DESC LIMIT 3;
\""
```

**通过标准**：
- `reservation.is_verified=1`, `status='in_progress'`, `verified_by='staff_X'`
- 新增一条 `gate_check_record`：`gate_id IS NULL`, `duration=预约时长`, `points_earned>0`

### 步骤 3.4 训练日历数据流通（意外之喜）

1. 手机 A 的会员 A 打开小程序
2. 去"训练日历"页面
3. 今天的格子应该有数据（打卡时长、积分）

**通过标准**：之前一直是 0 的数据现在有了（这是整个改造的最终目的）

> **阶段 3 通过 = 核心功能已上线可用。** 后续阶段是边缘场景。

## 阶段 4: 散客到店（无预约）

### 步骤 4.1 散客无邀请

1. 手机切换到会员 B（今天无任何预约）
2. 后台员工扫会员 B 的二维码
3. 应出现黄色警告框："该会员今日无可核销预约"
4. 下方有勾选框 + [记一条到店记录] 按钮
5. **不勾选**"由会员邀请而来"
6. 点 [记一条到店记录]

**通过标准**：toast "到店登记成功"，卡片清空

**验证 DB**：
```sql
SELECT * FROM gate_check_record WHERE member_id=<B的ID> ORDER BY id DESC LIMIT 1;
-- 应有一条：duration=0, gate_id=NULL, venue_id=刚才下拉选的场馆
```

### 步骤 4.2 散客有邀请（SS/SSS 会员 C）

1. 手机 B（会员 B）继续显示或重新打开二维码
2. 后台员工扫会员 B 的二维码 → 又看到"无预约"卡片
3. **勾选** "由会员邀请而来"
4. 应出现"请扫描邀请人的会员二维码"提示 + [扫邀请人码] 按钮
5. 点 [扫邀请人码] → 摄像头启动
6. 手机 C（会员 C）打开"我的会员码" → 员工扫
7. 应出现绿色提示 "✓ 已记录邀请人 token"
8. 点 [记一条到店记录]

**通过标准**：toast "到店登记成功"

**验证 DB**：
```sql
SELECT * FROM gate_check_record WHERE member_id=<B的ID> ORDER BY id DESC LIMIT 1;
-- 应有一条 duration=0 的新打卡

SELECT * FROM member_invitation WHERE inviter_id=<C的ID> AND invite_month='2026-04' ORDER BY id DESC LIMIT 1;
-- 应有一条 status='used', invite_code='WALKIN_xxx' 的新邀请记录
```

### 步骤 4.3 会员 C 配额递减

1. 后台重新扫会员 C 的二维码（或者 C 在小程序看邀请页）
2. 看"本月剩余邀请"数字

## 常见问题

### Q1: 扫码后一直显示"二维码已过期"
- 原因 1：小程序端的时钟跟服务器差距过大（>30s）
- 原因 2：扫的太慢，从小程序显示到员工扫完超过 30 秒
- 解决：在手机 B 打开小程序 → 快速切到会员 A 的手机展示 QR → 立即扫

### Q2: iPad Safari 扫码摄像头不工作
- 检查 iPadOS 版本是否 ≥ 14（html5-qrcode 对低版本支持不佳）
- 检查系统设置 → Safari → 摄像头访问权限
- 检查是否 HTTPS（yunlifang.cloud 已配 SSL）

### Q3: 核销后训练日历仍然为 0
- 立即刷新小程序可能有缓存，下拉刷新一下
- 检查 `gate_check_record` 表是否真的有数据（步骤 3.3 的 SQL）
- 检查 `Leaderboard` 是否需要手动 refresh：`POST /api/v1/checkin/leaderboard/refresh`

### Q4: 场馆下拉是空的
- 检查数据库 `venue` 表是否有数据
- 检查 `/api/v1/venues` 接口响应
- 检查 `venue.is_deleted=0` 的数据是否足够

### Q5: 散客邀请后 member_invitation 没数据
- 检查邀请人是否是 SS/SSS（level_code）
- 检查邀请人 `subscription_status='active'` 且 `member_expire_time > now()`
- 检查邀请人本月配额是否已用完
